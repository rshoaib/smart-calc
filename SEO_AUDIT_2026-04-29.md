# SmartCalc SEO Audit — April 29, 2026

**Site audited:** https://dailysmartcalc.com
**Source:** `C:\Projects\dailysmartcalc-next` (commit `3699f41`+, Next.js 16.1.6, React 19.2)
**Audit date:** 2026-04-29
**Scope:** Comprehensive (technical, on-page, content, competitive)

---

## Executive Summary

SmartCalc has a solid editorial foundation — 34 well-written, long-form blog posts with proper Article schema, a clean URL structure, a working sitemap and robots.txt, and FAQ schema on the homepage. The brand and content positioning is good.

But the site is being held back by **three critical SEO defects that are almost certainly suppressing rankings today:**

1. **The homepage and every primary calculator page bail out to client-side rendering.** Google receives an empty body containing only a loading spinner — none of the calculator UI, formulas, FAQs, or interlinks are in the initial HTML. This is the single biggest issue and likely explains weak rankings on commercial calculator queries.
2. **Every `<title>` tag is malformed** — the title template is being applied on top of titles that already include the brand, producing duplicates like `Mortgage Calculator | SmartCalc | SmartCalc`.
3. **All ~335 programmatic `/calculate/*` pages share an identical title and meta description** (the site default), because the route is a `'use client'` component that cannot export Next.js metadata. These pages are server-rendered with unique content, but search engines see them as 335 duplicate-title pages.

There are also a handful of broken internal links from blog posts to nonexistent calculator URLs (`/mortgage`, `/retirement`, `/compound-interest`, `/bmi`).

**Overall assessment: critical issues — but every one of them is fixable in a single afternoon.** The content quality is already there; once the rendering and metadata defects are resolved, the calculator pages should start gaining traction quickly.

## Critical Issues (fix this week)

### 1. Homepage and calculator pages render nothing on the server

**Severity: critical. Indexability and ranking impact.**

The live HTML for `/` and `/finance/mortgage` (and presumably every calculator page) contains:

```html
<template data-dgst="BAILOUT_TO_CLIENT_SIDE_RENDERING"></template>
<template data-dgst="BAILOUT_TO_CLIENT_SIDE_RENDERING"></template>
<div class="flex items-center justify-center min-h-[200px]">
  <div class="animate-spin rounded-full h-8 w-8 border-b-2 border-primary-500"></div>
</div>
```

That is the entire body. No H1, no calculator copy, no internal links. Compare this to `/calculate/20-percent-of-100`, which renders the full page in the initial HTML — proof that the rest of the site *can* SSR.

**Root cause:** the `Suspense` boundary in `app/layout.tsx` wraps `Providers` and `ClientLayout`, which call `useTranslation()` from `react-i18next` configured with `i18next-browser-languagedetector`. The browser language detector touches `window` during render, throws on the server, React unwinds the Suspense, and Next.js falls back to client-side rendering for that subtree. Pages that don't depend on a `useTranslation` call inside the rendered subtree (like `/calculate/[slug]` and `/blog`) survive; the home, finance, health, productivity hubs and every calculator page do not.

**Fix:**

- Configure i18next without a browser-only detector during SSR. Use `i18next-resources-to-backend` or hard-code the default language server-side, then upgrade to client detection on hydrate. The simplest fix: pass `detection: { order: [], caches: [] }` and rely on the URL/locale prop on the server.
- After the fix, view-source on the live homepage should contain the H1 (`Free Online Calculators for Finance, Health & Productivity`), the FAQ accordion content, and all the calculator card links — not a spinner.
- Verify with `curl https://dailysmartcalc.com/finance/mortgage | grep -i "PITI\|amortization"` — today this returns nothing.

**Expected impact:** high. This is gating ranking for every page that bails out, including the homepage. Google can render JS but it's slower, less reliable, and the content needs to be in the initial HTML to compete in commercial-intent SERPs against fully static sites like calculator.net.

### 2. Every page title is duplicated: "X Calculator | SmartCalc | SmartCalc"

**Severity: critical. CTR impact.**

`app/layout.tsx` declares `title.template = '%s | SmartCalc'`, which applies whenever a page sets a string title. Every calculator page sets `title: 'Mortgage Calculator | SmartCalc'` (the brand baked in), so the rendered title becomes `Mortgage Calculator | SmartCalc | SmartCalc` — confirmed in the live HTML for `/finance/mortgage`.

**Fix one of two ways:**

- **Recommended:** strip `| SmartCalc` from every per-page title and let the template add it once. Saves characters and is the idiomatic Next.js pattern.
  ```ts
  // app/finance/mortgage/page.tsx
  export const metadata: Metadata = {
    title: 'Free Mortgage Calculator with PMI, Taxes & Amortization',
    // template adds " | SmartCalc"
    ...
  };
  ```
- Or use `title: { absolute: '…' }` to opt out of the template per-page.

While you're touching every calculator's metadata, also rewrite the descriptions — see issue #4.

**Expected impact:** high. Duplicated brand in titles costs you ~10 characters of SERP real estate per result and looks unprofessional in the snippet.

### 3. 335 programmatic SEO pages share the same title and meta description

**Severity: critical. Duplicate-content risk; wasted programmatic-SEO investment.**

`app/calculate/[slug]/page.tsx` is marked `'use client'`. Next.js does not allow client components to export `metadata`, so every URL under `/calculate/*` falls back to the layout default:

```html
<title>SmartCalc — Free Finance & Health Calculators</title>
<meta name="description" content="Free online calculators for mortgage, investment, retirement, BMI, calories, and more...">
```

There are 335+ slugs covering percentage, tip, discount, age, salary, mortgage, BMI, and unit-conversion long-tail queries. Each has unique on-page content (the H1, formula, steps, and answer are correct in the rendered HTML), but search engines see 335 pages with identical titles. Google treats this as a strong duplicate-content signal and consolidates ranking onto one URL, killing the long-tail benefit.

The sitemap also explicitly **excludes** these pages (`Aggressive Sitemap Pruning: Explicitly excluding ...seoRoutes` in `app/sitemap.ts`), but the URLs still resolve and can be indexed via internal links from each other. So you have the worst of both worlds: indexable, internally-linked, and duplicate-titled.

**Fix:**

Convert `/calculate/[slug]` into a server component so it can export metadata. Use Next.js's recommended pattern:

```tsx
// app/calculate/[slug]/page.tsx — server component
import { allSeoPages } from '@/data/seoTemplates';

export async function generateStaticParams() {
  return allSeoPages.map(p => ({ slug: p.slug }));
}

export async function generateMetadata({ params }) {
  const { slug } = await params;
  const page = allSeoPages.find(p => p.slug === slug);
  if (!page) return {};
  return {
    title: page.title,                 // e.g. "What is 20% of 100?"
    description: page.description,
    alternates: { canonical: `/calculate/${slug}` },
  };
}

// then render <SeoCalculatorPage> the same way
```

Move the slug-lookup logic out of the client component. The current `useParams` + `redirect` pattern can be replaced by `notFound()` in the server component.

Then **add these URLs back to the sitemap**. With unique titles, descriptions, and the existing rich body content (formula, steps, FAQ), the long-tail query coverage is meaningful. Excluding 335 well-targeted pages from the sitemap because they had duplicate titles was treating the symptom; fixing the titles makes them a real asset.

**Expected impact:** high. Programmatic SEO is one of the highest-leverage growth levers a calculator site can have. Properly executed, these pages target hundreds of zero-competition long-tail queries (see competitive section).

## High-Priority Issues

### 6. No `<link rel="canonical">` on any page

Next.js will generate canonical URLs from `metadataBase` + `openGraph.url`, but a quick check of `/finance/mortgage`'s rendered HTML shows no `<link rel="canonical">` tag. Add `alternates: { canonical: '/finance/mortgage' }` (relative path resolves against `metadataBase`) to every calculator's metadata. This protects you from query-string and trailing-slash duplicate-content issues, especially given AdSense's tendency to attach tracking params.

### 7. `/finance`, `/health`, `/productivity` hub pages have no metadata

These pages are `'use client'` components with no `export const metadata`. They inherit only the layout default title (`SmartCalc — Free Finance & Health Calculators`) — meaning your three category hubs all share the homepage's title and description.

**Fix:** split each hub into a thin server `page.tsx` with metadata + a client component for the interactive pieces. Recommended titles:

- `/finance` → "Free Finance Calculators: Mortgage, Investment, Retirement, FIRE"
- `/health` → "Free Health Calculators: BMI, Calories, Macros, Heart Rate, Sleep"
- `/productivity` → "Free Productivity Tools: Pomodoro, Salary, GPA, Percentage"

### 8. Sitemap `lastmod` is `new Date()` for every URL on every revalidation

`app/sitemap.ts` uses `lastModified: new Date()` for static routes and tool routes. Every time the sitemap revalidates, every URL appears to have just been updated. Google deprioritizes crawling sites that lie about update frequency.

**Fix:** stamp tool pages with the build date (read once at module load, not per request), and use the actual `mtime` of the markdown file for blog posts (you have `frontmatter.date` and optional `updated_at` already — use `updated_at || date`).

```ts
import { statSync } from 'fs';
const buildTime = new Date(); // evaluated once at module load

// blog routes:
const blogRoutes = blogPosts.map((post) => ({
  url: `${baseUrl}/blog/${post.slug}`,
  lastModified: post.updatedAt ?? post.date,
  ...
}));
```

### 9. Calculator pages have no FAQ schema even though FAQ-quality copy exists

The homepage has FAQ schema (good), but individual calculator pages have rich Q&A-shaped content in their ClientComponents and no FAQ JSON-LD. Same for blog posts that contain "Frequently Asked Questions" sections in markdown. Adding FAQPage schema is one of the easiest ways to win SERP real estate (collapsible Q&As under your result).

**Fix:** add a `<script type="application/ld+json">` with `FAQPage` schema in each calculator's server `page.tsx`, populated from a small `faqs` array. Mirror what `app/page.tsx` already does for the homepage.

### 10. No `BreadcrumbList` schema or visible breadcrumbs

Pages have category structure (`/finance/mortgage`, `/health/bmi`) but no breadcrumb UI and no `BreadcrumbList` JSON-LD. Breadcrumb schema is a near-free win — Google often shows breadcrumbs in the SERP instead of the URL, which improves CTR.

**Fix:** add a small `<Breadcrumbs>` component (Home → Finance → Mortgage Calculator) at the top of every tool page, and emit `BreadcrumbList` JSON-LD alongside.

### 11. `<html lang="en">` is hardcoded but Spanish locale exists

`locales/es/common.json` is wired up via i18next, and there's a language toggle, but the `<html lang>` attribute never changes from `en`, and there's no `hreflang` markup. If you're not actively maintaining Spanish content, **remove the toggle** rather than half-shipping it — incomplete localization confuses crawlers more than it helps. If you intend to maintain `/es/*` pages, add hreflang alternates and locale-specific URLs.

### 12. Blog posts lack `dateModified` in Article schema

`app/blog/[slug]/page.tsx` emits `datePublished` but no `dateModified`. Use `frontmatter.updated_at` if present, otherwise fall back to `date`. Google's freshness signals weigh this for evergreen guides.

## Medium-Priority Issues

### 13. Open Graph image is missing on calculator and blog list pages

The rendered HTML for `/finance/mortgage` has no `og:image` tag, even though `og-image.png` exists at the root. Each calculator's metadata block only sets title/description/url/type for openGraph. Blog list page has no openGraph block at all.

**Fix:** either set a fallback `images: [{ url: '/og-image.png' }]` in each calculator's metadata, or hoist a default into `metadataBase` and let Next.js cascade. Better: generate per-tool OG images (already done for blog posts via `buildBlogHeroSvg` — extend the same pattern to calculators).

### 14. Sitemap reports `priority: 0.3` for legal pages that aren't crawled anyway

`/privacy`, `/terms`, and `/about` are not ranking pages. Listing them with priority 0.3 is harmless but distracting; consider whether `/about` and `/contact` should be `0.4`–`0.5` (they're brand/E-E-A-T pages).

### 15. AdSense script loaded in `<head>` with `afterInteractive`

Loading the AdSense script as `Script` in the document `<head>` with `strategy="afterInteractive"` is fine, but the `<link rel="preload" href="…adsbygoogle.js">` in the rendered HTML is redundant with the async `<script>` and pulls in render-blocking work earlier than needed. Recommend `strategy="lazyOnload"` for AdSense unless you're seeing ad-fill issues.

### 16. Anchor text on internal sidebar links is generic

The right rail says things like "Mortgage Calculator", "Investment Return", "Auto Loan Calculator" — fine. But the footer links say "Free Mortgage Calculator", "ROI Calculator", "BMI Calculator", "Word Counter", "Typing Speed Test" — only 5 links. Expand the footer to ~15 internal links covering the highest-priority calculators with descriptive anchors. Footer links don't pass much PageRank but they help with internal-link diversity for crawlers.

### 17. No image `alt` attributes on the dynamic blog hero SVGs

`buildBlogHeroSvg` is injected via `dangerouslySetInnerHTML`. The wrapping `<div>` has no `aria-label` or `role="img"`. Wrap the SVG container with `<div role="img" aria-label={post.title}>` so screen readers announce it (and Google can use the label as image context).

### 18. The blog post page doesn't link to other calculator pages outside the related tool

A blog post about mortgages should link to `/finance/auto-loan`, `/finance/rent-vs-buy`, `/finance/investment` (compound interest is mortgage's mirror). Currently the Read Next section uses three category-aware suggestions, but the markdown body has no programmatic interlinking. Adding 2–3 contextual cross-links per post would meaningfully improve internal linking depth.

## Low-Priority / Nice to Have

### 19. Use `WebSite` schema with `SearchAction`
Add a `potentialAction` to the WebSite JSON-LD on the homepage describing your search box (you don't have one — but if you add one, this enables a sitelink search box in the SERP).

### 20. Use `HowTo` schema on calculator pages where relevant
The mortgage, BMI, salary-hourly, percentage, and tip calculators all walk through formulaic steps. `HowTo` schema is appropriate and may surface a richer SERP card.

### 21. Add a `<meta name="robots" content="max-image-preview:large">`
Improves image preview size in Google Discover, which matters for blog posts with strong hero SVGs.

### 22. Trailing-slash inconsistency in `og:url`
`app/layout.tsx` declares `https://dailysmartcalc.com/` (trailing slash) for the homepage canonical, but tool pages use `https://dailysmartcalc.com/finance/mortgage` (no trailing slash). Vercel handles this gracefully but pick a convention and stick to it across canonical, openGraph.url, and the sitemap.

### 23. Dead Supabase code from the migration
Per project memory, the Supabase data layer was ripped out (commit `3699f41`). The sitemap still has a comment "Blog posts from Supabase" — cosmetic, not functional, but worth a quick clean-up while you're in there.
